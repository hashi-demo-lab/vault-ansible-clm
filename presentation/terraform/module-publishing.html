Situation

Modules need to reach consumers reliably

Platform teams invest in building validated, tested Terraform modules. The Private Module Registry is the standard distribution channel — but the journey from “code on a branch” to “consumable registry entry” has more moving parts than the docs suggest.

The Goal
Merge a PR, get a versioned module in PMR
Automated, repeatable, with proper metadata — source link, branch, commit SHA — visible in the registry for auditability.
Complication

The “last mile” is harder than it looks

Publishing a module to PMR seems straightforward — connect a repo, push a tag. In practice, there are hidden requirements around naming, token permissions, version immutability, and VCS configuration that cause silent failures or modules stuck in pending.

🔌

VCS Configuration

The publishing mechanism (tag vs branch) is permanent once the module is created. The repo must follow terraform-{provider}-{name} naming. Renaming later breaks ingress.

🔒

Token Scoping

CI tokens need Manage Modules at the org level — not workspace permissions. A read-capable token that can’t write leads to confusing partial failures.

📌

Version Immutability

Published versions cannot be overwritten. READMEs are frozen per version. A bad release means publishing a patch — there is no “replace in place.”

Understanding the Options

Three publishing models

Tag-Based (VCS)

Push a semver tag → PMR auto-detects and ingests.

Fully automatic — push tag, version appears
Source, branch, commit in PMR UI
Tag prefix support for monorepos
No built-in PMR test support
Non-semver tags silently ignored
Best for CI/CD automation

Branch-Based (VCS)

PMR tracks a branch. Versions created explicitly via API or UI.

Built-in PMR testing on every commit
Remote CLI: terraform test -cloud-run
Source, branch, commit in PMR UI
Merge to main does NOT auto-publish
Must create each version explicitly
Best for dev with testing needs

API-Driven (No VCS)

Upload tarballs via API. No VCS connection required.

No repo naming requirement
Works in air-gapped environments
Full control over published content
No source/branch/commit in PMR UI
Must package & upload tarball every time
When VCS isn’t an option

Answer — Automation

Two-workflow CI/CD pipeline

module_validate.yml — Triggered on PR

Runs when .tf, .tfvars, or .tftest.hcl files change. Performs the full quality gate before code can be merged.

module_release.yml — Triggered on merge

Runs on pull_request: closed (not push — this gives access to PR labels). Handles version calculation, publishing, tagging, and release creation.

Why pull_request: closed?
A push-triggered workflow can’t read PR labels. Using the closed event with a merged == true guard gives you both the merge commit and the label context for semver determination.
Answer — Pipeline Considerations

What to consider when automating

Semver label enforcement

Every PR must carry exactly one label: semver:patch, semver:minor, or semver:major. The validation workflow should fail fast if no label is present — before burning runner minutes on tests.

Unit tests should be blocking

Don’t use continue-on-error: true for tests in CI. A PR that merges with failing tests will publish a broken version. Make tests a hard gate.

terraform-docs auto-commit

Use terraform-docs/gh-actions with git-push: true to keep inputs/outputs documentation current.

Version source of truth

Query the PMR API for the current latest version rather than relying on git tags. PMR is the authoritative source — tags and PMR can drift if either is managed manually.

Example validation in parallel

Run terraform init and validate on each example directory as a separate matrix job. This parallelizes the work and catches broken examples before merge.

Branch protection matters

Require status checks to pass before merge. If admins can bypass checks, a PR without a semver label can merge — and the release workflow has no way to determine the version bump.

Things to Know

Gotchas that aren’t obvious from the docs

01

Versions are immutable

Once published, a version cannot be overwritten (HTTP 422). To fix a bad release, publish a new patch or delete the version via API first.

Tip: Use deprecation/revocation instead of deletion when possible
02

README is frozen per version

PMR captures the README at ingestion time. Fixing the README without bumping the version changes nothing in the registry.

Tip: Run terraform-docs and review prose before every release
03

Publishing mechanism is permanent

Tag-based or branch-based is frozen at creation. Switching requires deleting the module entirely and recreating it.

Tip: Decide your model before creating the module
04

Repo renames break VCS

HCP Terraform caches the repo identifier at creation. After a rename, ingress can fail for hours. HashiCorp recommends against renaming repos with published modules.

Tip: Name the repo correctly before creating the VCS module
05

Non-semver tags are silent

Tags that don’t match vX.Y.Z or X.Y.Z are silently skipped. No error, no log. Module stays in pending with no feedback.

Tip: Use strict semver — v1.0.0 or 1.0.0, nothing else
06

Monorepos need tag prefixes

Without prefixes, pushing v1.0.0 publishes for every module in the repo. Use networking-v1.0.0 to isolate. Registry strips the prefix automatically.

Tip: One module per repo is simpler unless modules are coupled
07

Deleting the last version cascades

Deleting the last version of a provider auto-deletes the provider. Deleting the last provider auto-deletes the module. Cascading cleanup.

Tip: Use version deprecation instead of deletion
08

terraform-docs only manages its markers

Content between BEGIN_TF_DOCS/END_TF_DOCS is auto-generated. Everything above is static prose you must maintain yourself.

Tip: Write module-focused prose above the markers before first publish
Best Practices

Token security and CI/CD hygiene

Use dedicated team tokens

Create a team like ci-module-publishers with only Manage Modules. Avoid org tokens (over-privileged) and user tokens (tied to individuals). Teams can hold multiple valid tokens for zero-downtime rotation.

Variables vs secrets

Module coordinates (TFE_ORG, TFE_MODULE, TFE_PROVIDER) as repo variables for log visibility. TFE_TOKEN as a secret. Never echo tokens in CI output.

Permissions 
Token Type       Publish?  Notes
Team token       Yes       Recommended for CI
User token       Yes       Tied to individual
Org token        Yes       Over-privileged
Agent token      No        Agent pools only
Common failure
Read succeeds, write fails
A Traverse token can query existing versions but creating a new version needs Manage Modules. Version calculator works, publish step returns 404.
Best Practices

Your README is your module’s first impression

PMR displays the root README as the module’s documentation page. The README is captured at ingestion time and frozen for that version.

terraform-docs integration
Auto-generate inputs, outputs, and resources
Use terraform-docs with BEGIN_TF_DOCS / END_TF_DOCS markers. The content between these markers is auto-generated and always current. Run it in CI with git-push: true to auto-commit updates to every PR.

What goes above the markers (manual)

Module title, one-line description, features list, basic and advanced usage examples with the PMR source path, prerequisites (provider version, required cloud resources), and security defaults. terraform-docs never touches this content — you maintain it.

Remember
README is frozen per version
Fixing the README after publishing does nothing in PMR. You must publish a new version for consumers to see the update. Always review the full README before every release.

Summary

Key takeaways

  • 01
    Choose your publishing model deliberatelyTag-based auto-publishes on tag push. Branch-based requires explicit version creation but supports PMR testing. API-driven gives full control without VCS. The choice is permanent.
  • 02
    Name the repo correctly firstFollow terraform-{provider}-{name} for VCS modules. Renaming later breaks VCS caching. Get it right before creating the module in PMR.
  • 03
    Get token permissions right on day oneTeam tokens with Manage Modules. Verify the token can both read and write. HTTP 404 on publish means wrong permissions, not wrong URL.
  • 04
    Versions and READMEs are frozen once publishedNo overwrites. Run terraform-docs and verify prose README before every release. A bad release means a new patch version.
  • 05
    Automate with PR-merge triggersUse pull_request: closed (not push) for label access. Enforce semver labels via required status checks to prevent unrecoverable failures.
  • 06
    Test before you shipMake unit tests blocking in CI. Branch-based modules can additionally use PMR’s built-in testing with terraform test -cloud-run.
