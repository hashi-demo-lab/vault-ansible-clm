The Goal
Validate configs at Git commit & before deploy
Using Terraform input validation rules, we can already validate YAML structure. Policies are the remaining gap.

What is yamldecode()?

A built-in Terraform function that converts a YAML string into a native Terraform object (maps, lists, strings). This lets Terraform inspect and validate the data inside YAML files at plan time — no external tools needed.

Situation — Current State

Policies today live as raw HCL strings

Consumer teams embed Vault policy HCL directly in their YAML configuration files via inline_policies. The HCL is a plain string — Terraform passes it through without inspection.

YAML team_config.yaml 
jwt_roles_access:
  - role_name: my_role1
    jwt_type: gitlab
    inline_policies: |
      # This is an opaque string to Terraform
      path "/kv/data/secret1" {
        capabilities = ["read"]
      }
  - role_name: my_role2
    jwt_type: gitlab
    inline_policies: |
      path "/kv/data/secret2" {
        capabilities = ["read"]
      }
The Problem
Terraform cannot see inside this string
After yamldecode(), the policy is just a string like "path \"/kv/data/secret1\" { capabilities = [\"read\"] }". Terraform has no way to check if the HCL is valid, if capabilities are real, or if paths are allowed.

What is an HCL string?

HCL (HashiCorp Configuration Language) is the syntax Vault uses for ACL policies. When embedded in YAML via the | pipe operator, it becomes a multi-line string. Terraform sees it as plain text — it doesn't parse the HCL grammar inside.

Why does this matter?

Errors in the policy string — typos like "reade" instead of "read", or dangerous paths like sys/auth/* — won't be caught until Vault tries to apply them, which may be too late.

Complication

Terraform has no hcldecode()

Terraform provides decode functions for YAML and JSON, but nothing for HCL. Policy strings remain opaque — a blind spot in our validation pipeline.

What We Want to Catch
Three categories of policy errors — all before Vault sees them
Syntax errors — malformed HCL or JSON
Structural errors — missing capabilities, invalid nesting
Data errors — invalid capabilities like "reade", disallowed path prefixes like "sys/auth/*"
Complication — Evaluated Approaches

Answer

Restructure the data model entirely

Instead of parsing HCL inside Terraform, don't pass HCL strings through Terraform at all. Define policies as structured YAML objects that Terraform can natively type-check.

YAML team_example.yaml — new format 
team_name: "example-team"

jwt_roles:
  - role_name: example-app1
    jwt_type: gitlab
    bound_claims:
      namespace_id: "12345"
    policy_rules:
      - path: "kv/data/teams/example-team/*"
        capabilities: ["read", "list"]
      - path: "kv/metadata/teams/example-team/*"
        capabilities: ["list"]
Key Insight
Policies become data, not strings
After yamldecode(), each policy rule is a Terraform object with typed fields: path is a string, capabilities is a list. Terraform can validate every field natively.

What changed?

Instead of inline_policies: | (a raw HCL string), we now have policy_rules: (a list of objects). Each object has a path and capabilities array — structured data that Terraform understands.

But who generates the actual HCL for Vault?

The vault_policy_document data source takes structured input and generates valid HCL policy text. Terraform handles the HCL generation — consumers never write it.

Answer — Validation Layer

Five validation rules, zero external tools

Terraform variable validation blocks run during plan, before any API calls. Each rule uses native Terraform functions — contains(), startswith(), alltrue().

Terraform modules/consumer_validation/variables.tf 
# Validate: capabilities must be from Vault's allowed set
validation {
  condition = alltrue([
    for p in var.consumer_config_paths : alltrue(flatten([
      for role in yamldecode(file("${path.module}/${p}"))["jwt_roles"] : [
        for rule in role["policy_rules"] : [
          for cap in rule["capabilities"] :
          contains(["create","read","update","delete","list","sudo","deny"], cap)
        ]
      ]
    ]))
  ])
  error_message = format(
    "Invalid capability detected. Allowed: create, read, ... Found: %s",
    join(", ", distinct(flatten([/* identifies the bad values */])))
  )
}

Structural

Every role has ≥1 rule, every rule has ≥1 capability

Capability Check

All capabilities from Vault's allowed set (create, read, update…)

Path Prefix

Paths must start with allowed prefixes (kv/data/, kv/metadata/…)

Security

No sudo on wildcard paths — prevents privilege escalation

Naming

Role names must be 3–64 chars, alphanumeric with hyphens

How do validation blocks work?

Each validation block on a variable has a condition (must be true) and an error_message. They run during terraform plan — before resources are created. If any condition is false, the plan fails immediately with the error message. Multiple validation blocks on the same variable run in order.

Answer — HCL Generation

vault_policy_document generates the HCL for you

The vault_policy_document data source is the bridge between structured data and Vault. It takes rule blocks as input and outputs valid HCL policy text via the .hcl attribute.

Terraform modules/consumer_validation/main.tf 
# Generate Vault policy HCL from structured rules
data "vault_policy_document" "consumer_policies" {
  for_each = local.all_roles

  dynamic "rule" {
    for_each = each.value.policy_rules
    content {
      path         = rule.value["path"]
      capabilities = rule.value["capabilities"]
    }
  }
}

# Create the actual Vault policies
resource "vault_policy" "consumer_policies" {
  for_each = local.all_roles

  name   = each.key
  policy = data.vault_policy_document.consumer_policies[each.key].hcl
}

What is a data source?

A Terraform data source reads or computes information without creating infrastructure. vault_policy_document runs locally during plan — it doesn't call the Vault API. It converts structured input into the exact HCL format Vault expects.

What is dynamic "rule"?

The dynamic block generates repeated nested blocks from a list. Instead of writing each rule {} block by hand, it loops over policy_rules from the YAML and creates one rule per entry. This is how structured YAML data flows into the policy document.

Double Validation
vault_policy_document also validates capabilities internally
Even if someone bypassed our variable validation, this data source would reject invalid capabilities — a second safety layer for free.

Summary

The structured approach eliminates the parsing gap

  • 01
    Restructure, don't parse Replace raw HCL strings with structured YAML objects (path + capabilities). Terraform validates data it understands — don't ask it to parse a language it can't read.
  • 02
    Variable validation runs at plan time Five validation rules catch syntax, structural, data, and security errors before any Vault API call. Each rule uses only native Terraform functions.
  • 03
    vault_policy_document bridges the gap This data source converts structured input into valid HCL policy text. It also validates capabilities internally — a second safety layer for free.
  • 04
    Zero external dependencies No Python scripts, no hcl2json, no shell exec. Pure Terraform. Works identically in every CI/CD environment.
  • 05
    Migration is a one-time cost Existing HCL policies can be converted to the structured format with a simple script. New consumers never write HCL — they define structured rules and Terraform handles the rest.
